# Feedback Loop

Use this file when the recommendation is important enough to deserve a second pass.

The goal is to separate creator momentum from judgment. Do not merely reword the same answer. Re-evaluate it.

## Pass 1: Create

Draft the recommendation using `SKILL.md` and `decision-playbook.md`.

The draft should include:
- recommendation
- why it wins now
- assumptions
- risks
- metrics
- next step

## Pass 2: Judge

Reread only:
- the user request
- the current draft
- this file

Then score the draft against these checks:

- Is the decision itself clear?
- Did it use the right problem checklist from `problem-map.md`?
- Does the recommendation match the startup stage?
- Does it identify the real bottleneck instead of polishing symptoms?
- Are assumptions explicit?
- Are real alternatives considered?
- Is the advice specific to this business instead of generic startup talk?
- Does the pricing or GTM recommendation fit the team, support load, and price point?
- Does it respect distribution reality instead of inventing a channel edge?
- Is the next move small enough to run and strong enough to teach?
- Are success and failure measurable?
- Would the founder know what to do next week?

## Failure Modes

Revise if the draft:
- says "it depends" without resolving the dependency
- gives broad brainstorming instead of a decision
- ignores runway, team capacity, or distribution reality
- recommends freemium, low pricing, or a heavy channel without checking the prerequisites
- recommends several priorities without choosing one
- optimizes vanity metrics over retained value or learning
- sounds certain without evidence
- skips the measurement or review window

## Decision Rules

- `accept`: recommendation is stage-aware, specific, instrumented, and action-ready
- `minor revision`: core recommendation is sound, but assumptions, risks, or metrics need tightening
- `major revision`: the draft solved the wrong problem, missed the bottleneck, or stayed generic

## Tighten the Revision

When revising:
- change the decision, not just the wording, if the critique requires it
- cut extra options if they create fog
- sharpen the first move until it can actually be executed
- make the review window and success signal explicit

Stop when the answer is decision-useful, not when it is maximally verbose.
