# Simplicity Before Surface Area

Use when:
- the product feels bloated
- support is rising
- the founder is tempted to add more features before clarifying the core path

Decision types:
- roadmap
- support
- scope control

What must be true:
- the product already has enough surface area that complexity imposes support or maintenance cost

Why it works:
- simplicity compounds for small teams by reducing onboarding friction, support load, and founder stress

What to copy:
- fix confusing UX before expanding docs
- make every feature earn its place through retention, revenue, or decisive learning
- remove the need for explanatory content by making the core path more intuitive

What not to copy:
- hiding poor UX behind more documentation
- using how-to content to excuse a confusing core flow
- broadening the product because edge cases are loud

Failure modes:
- complexity tax becoming a permanent operating burden
- admired feature breadth masking weak core value

What to measure:
- repeat support questions
- time to first value
- usage of new versus core features
- churn or dropoff tied to complexity
