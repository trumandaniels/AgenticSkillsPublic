# Taxonomy

This taxonomy is seeded from the processed books in `source-map.md`. Keep it as a routing aid, not a prose summary.

| Category | Triggering symptoms | Required context | Common bad move | Source anchors |
|---|---|---|---|---|
| naming | unclear variables/functions/classes; names too short or too long; domain terms absent from code | symbol scope, use frequency, line shape, domain terms, expert language | renaming public symbols without consumer analysis | `book1:chapter-7-simplicity`, `book2:chapter-2-ubiquitous-language`, `book2:chapter-10-supple-design` |
| functions | long pieces, mixed responsibilities, hard-to-change chunks, side effects mixed with queries | tests, call sites, variable lifetime, behavior boundaries, command/query split | splitting mechanically without making pieces simpler | `book1:chapter-7-simplicity`, `book1:chapter-8-complexity`, `book2:chapter-10-supple-design` |
| abstractions | speculative flexibility, over-generic APIs, wrapper decisions, unclear entity/value/service choice, domain packages grouped by technical type | known present need, real variants, identity, lifecycle, user-facing specificity, conceptual cohesion | future-proofing from guesses, forcing every concept into an entity, or packaging by implementation mechanics when domain concepts should lead | `book1:chapter-5-change`, `book1:chapter-8-complexity`, `book2:chapter-5-model-elements`, `book2:chapter-5-modules`, `book2:chapter-6-lifecycle` |
| duplication | copy/paste code, repeated strings/config/rules, duplicated domain concepts across contexts | semantic sameness, model context, change coupling, source of truth | merging similar-looking rules that may diverge | `book1:chapter-6-defects`, `book2:chapter-14-model-integrity` |
| comments | comments restate code or missing rationale | code clarity, hidden constraints, likely future readers | using comments to excuse unclear code | `book1:chapter-7-simplicity` |
| error-handling | vague errors, lost user/domain meaning, too-generic handling, overload behavior hidden inside generic failures | actual failure modes, user need, boundary layer, retry/backoff/throttle semantics | hiding specific cases behind generic bytes/errors or returning success during overload without a policy | `book1:chapter-5-change`, `book3:rate-limiter` |
| testability | changed behavior lacks proof; tests are vague or absent; context translation is unverified; scale or failure claims are unobserved | intended behavior, model assertions, environments, observability, automation, latency/throughput/failure signals | saying it works without trying it or adding scale mechanisms without measuring the bottleneck | `book1:chapter-9-testing`, `book2:chapter-10-supple-design`, `book2:chapter-14-model-integrity`, `book3:system-design-process`, `book3:distributed-characteristics` |
| api-boundaries | external complexity, database/platform variance, public contracts, legacy integration, bounded-context seams, cache/storage/partition choices, queues, load balancing, proxying, realtime transports | consumers, current need, compatibility, model boundary, translation ownership, access pattern, consistency need, hot key risk, delivery semantics, connection shape | sprinkling special cases everywhere, importing foreign model concepts directly, scaling by mechanism before clarifying requirements, or adding async/realtime machinery without a contract | `book1:chapter-8-complexity`, `book2:chapter-4-layered-architecture`, `book2:chapter-14-model-integrity`, `book3:system-design-process`, `book3:caching`, `book3:sharding-partitioning`, `book3:sql-nosql-cap`, `book3:load-balancing`, `book3:message-queues`, `book3:realtime-transports` |
